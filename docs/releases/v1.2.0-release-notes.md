# Aurora Edge Runtime v1.2.0 Release Notes

**Release Date:** 2026-03-23
**Version:** 1.2.0
**Status:** Major Release

---

## 🎯 Overview

Aurora Edge Runtime v1.2.0 is a **major release** focusing on **dual-process synchronization** and **trigger collection accuracy**. This release resolves critical issues affecting long-running missions and data collection reliability in dual-process mode (DCP + robot_sim).

### Why This Matters

Previous versions experienced:
- ❌ Odometry staleness after ~30 seconds, causing navigation failures
- ❌ Triggers firing at planned waypoint coordinates instead of actual robot position
- ❌ Path discontinuity causing robot jumps and timeout errors
- ❌ Invalid training data due to position inaccuracies

v1.2.0 addresses all these issues with:
- ✅ **99.6% improvement** in odometry freshness (>5000ms → ~19ms)
- ✅ **Accurate triggers** based on actual robot position
- ✅ **Smooth path transitions** between planning cycles
- ✅ **Valid training data** for reinforcement learning

---

## 🚀 Key Improvements

### 1. Odometry Synchronization (Critical)

**Problem:**
```
After ~30 seconds of operation:
- Odometry age: >5000ms
- robot_sim stops updating odometry
- DCP times out waiting for waypoints
- Mission fails
```

**Solution:**
- Implemented **condition variable synchronization** for waypoint waiting
- Added `waypoint_wait_time_` parameter (default: 3s) for DCP coordination
- Thread-safe state tracking with mutex protection
- robot_sim continues odometry updates while waiting for DCP

**Results:**
```
✅ Odometry age: ~19ms (steady state)
✅ Robot continues through all waypoints
✅ No more timeout errors
✅ Reliable long-running missions
```

### 2. Trigger Position Accuracy (Critical)

**Problem:**
```
Trigger evaluation used planned waypoint coordinates:
- Log: "Trigger accepted at (0.00, 0.00), step distance: 6.21 m"
- Reality: Robot was at (6.00, 4.50)
- Impact: Invalid training data, inaccurate rewards
```

**Solution:**
- Added **robot position tracking** to `TriggerManager`
- Implemented `updateRobotPosition(x, y)` method
- Wire up odometry callback at 50Hz to feed actual position
- Modified `shouldTrigger()` to use actual robot odometry position

**Results:**
```
✅ Triggers evaluate at actual robot position
✅ Accurate step distance calculations
✅ Valid training data for RL
✅ Correct reward computation
```

### 3. Path Continuity (Important)

**Problem:**
```
Between planning cycles:
- robot_sim at position A
- New path starts at position B
- Robot jumps from A to B
- Timeout errors occur
```

**Solution:**
- Add robot's current position to beginning of each new path
- Implement `/robot/get_position` service call
- Smooth transitions between cycles

**Results:**
```
✅ No sudden robot jumps
✅ Continuous motion through all cycles
✅ Reduced timeout errors
```

## 🔄 Migration Guide

### For Users

#### 1. Update Dependencies

No new dependencies required. All changes are internal improvements.

#### 2. Configuration Updates

**Optional:** Add waypoint wait time to `config/planner_weights.yaml`:

```yaml
# Dual-process synchronization (optional, default: 3.0)
waypoint_wait_time: 3.0  # seconds to wait at each waypoint
```

#### 3. Update Monitoring

Update your monitoring to expect:
- **Odometry age:** Should stay < 100ms (previously >5000ms)
- **Trigger positions:** Will match actual robot position (not planned waypoints)
- **Mission duration:** Can now run indefinitely without odometry staleness

#### 4. Verify Operation

```bash
# Start dual-process mode
./launch/launch_dual_process.sh

# Monitor odometry age (should stay < 100ms)
ros2 topic echo /robot/odom --field header.stamp

# Check trigger positions (should match actual robot location)
ros2 topic echo /collection/triggers
```

### For Developers

#### API Changes

**New Methods:**
```cpp
// TriggerManager
void updateRobotPosition(double x, double y);
Point getRobotPosition() const;

// DataCollectionExecutor
TriggerManager* getTriggerManager();
```

**Thread Safety:**
- `TriggerManager::robot_position_` is now protected by `robot_position_mutex_`
- `robot_simulator_v2::waypoint_mutex_` protects waypoint state
- Always use mutex when accessing shared state

#### Testing

Update your tests to verify:
1. Odometry age stays < 100ms throughout 5+ minute missions
2. Trigger coordinates match actual robot odometry position
3. No path discontinuities between planning cycles
4. Thread safety under concurrent access

## ⚠️ Breaking Changes

**None.** All changes are backward compatible.

### Behavior Changes

While not breaking changes, be aware of these behavior changes:

1. **Trigger positions** now use actual robot position (not planned waypoints)
   - Impact: More accurate triggers, valid training data
   - Action: Update any code expecting planned waypoint coordinates

2. **Odometry updates** continue during waypoint waiting
   - Impact: Odometry stays fresh indefinitely
   - Action: Update monitoring thresholds

## 📝 Known Issues

### Unresolved

1. **ROS2 Service Discovery**
   - Symptom: Service calls timeout occasionally
   - Impact: Non-blocking (core functionality works)
   - Priority: Medium
   - Workaround: Auto-retry handles most cases

2. **FastDDS Shared Memory**
   - Symptom: Requires cleanup after crashes
   - Impact: Manual intervention needed
   - Priority: Low
   - Workaround: Run `rm -rf /dev/shm/*` if issues occur

## 🚢 Deployment

### Recommended Deployment Steps

1. **Pre-deployment:**
   ```bash
   # Backup current configuration
   cp -r config/ config.backup/

   # Review release notes
   cat docs/releases/v1.2.0-release-notes.md
   ```

2. **Deployment:**
   ```bash
   # Build new version
   cd build && make -j8

   # Test in staging environment
   ./launch/launch_dual_process.sh
   ```

3. **Verification:**
   ```bash
   # Monitor odometry age (should stay < 100ms)
   watch -n 1 'ros2 topic echo /robot/odom --field header.stamp'

   # Check trigger positions
   ros2 topic echo /collection/triggers
   ```

4. **Production Rollout:**
   ```bash
   # Stop old version
   sudo systemctl stop aer

   # Deploy new version
   # ... (your deployment process)

   # Start new version
   sudo systemctl start aer

   # Verify operation
   sudo systemctl status aer
   ```
