## DDR-001: 端云协同架构

### 上下文

自动驾驶数据采集需要处理大量传感器数据，同时进行智能路径规划。边缘设备计算能力有限，无法在本地完成深度学习训练。

### 决策

采用**端云协同架构**：

| 端侧 (Edge) | 云端 (Cloud) |
|------------|-------------|
| 模型推理 (<10ms) | 模型训练 |
| 数据采集 | 数据分析 |
| 轻量级反馈 | 策略优化 |
| 实时控制 | 批量处理 |

### 原因

1. **计算资源优化**：端侧专注轻量级推理，云端承担重计算
2. **带宽效率**：只传输元数据 (~20B) 而非完整数据 (~250B)
3. **实时性**：端侧推理延迟 <10ms，满足实时控制需求
4. **可扩展性**：云端可方便扩展更多训练资源

### 权衡

| 优点 | 缺点 |
|------|------|
| 降低端侧资源需求 | 依赖网络连接 |
| 支持模型持续优化 | 云端故障时无法更新 |
| 统一多设备策略 | 延迟可能影响反馈 |

## DDR-002: PPO 算法选择

### 上下文

需要选择合适的强化学习算法用于路径规划。候选算法包括 DQN、DDPG、A3C、PPO 等。

### 决策

采用 **PPO (Proximal Policy Optimization)** 算法。

### 原因

1. **稳定性**：PPO 通过裁剪机制保证策略更新稳定
2. **样本效率**：相比传统策略梯度方法有更好的样本效率
3. **易调试**：超参数相对容易调节
4. **离散/连续兼容**：支持离散和连续动作空间

### 对比

| 算法 | 样本效率 | 稳定性 | 实现复杂度 |
|------|----------|--------|-----------|
| DQN | 低 | 中 | 低 |
| DDPG | 中 | 低 | 高 |
| A3C | 中 | 中 | 中 |
| **PPO** | **高** | **高** | **中** |

## DDR-003: ONNX 模型格式

### 上下文

需要在云端训练的模型部署到边缘端。候选格式包括 PyTorch 模型、TensorFlow SavedModel、ONNX 等。

### 决策

采用 **ONNX (Open Neural Network Exchange)** 格式。

### 原因

1. **跨平台**：支持多种框架和硬件
2. **标准化**：工业标准，生态完善
3. **性能**：ONNX Runtime 针对推理优化
4. **版本兼容**：支持 opset 17，向后兼容

### 技术细节

```cpp
// ONNX Runtime 初始化
Ort::Env env(ORT_LOGGING_LEVEL_WARNING, "dcp");
Ort::SessionOptions options;
options.SetIntraOpNumThreads(1);
options.SetGraphOptimizationLevel(GraphOptimizationLevel::ORT_ENABLE_ALL);

// 推理性能
// - 模型加载: < 100ms
// - 单次推理: < 10ms
// - 内存占用: < 50MB
```

## DDR-004: 环形缓冲区采集

### 上下文

数据采集需要记录触发前后的完整数据。传统方式是持续记录并删除旧文件，但这会导致关键数据丢失。

### 决策

采用**环形缓冲区 (Ring Buffer)** + **触发保存**机制。

### 原因

1. **完整性**：保证触发前数据不丢失 (默认 15s)
2. **高效性**：内存操作，无磁盘 I/O
3. **灵活配置**：可调整前后缓存时长

### 架构

```
环形缓冲区结构:
┌────────────────────────────────────┐
│ [Pre-trigger: 15s] │ [Post: 5s]   │
└────────────────────────────────────┘
     ↑ 触发时刻
     ↓
    保存整个缓冲区
```

### 配置

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `forward_duration` | 15s | 触发前缓存时长 |
| `backward_duration` | 5s | 触发后录制时长 |
| `mode` | ring | 缓存模式 |

## DDR-005: 步态触发采集

### 上下文

双足机器人数据采集传统方式是在规划路径点触发采集。但实际行走中，足端落地点与规划点存在偏差。

### 决策

采用**基于实际足端落地的触发**机制，而非规划路径点。

### 原则

1. **实际落地优先**：采集点基于足端实际落点
2. **稳定期采集**：只在双脚支撑稳定期触发
3. **步长约束**：确保最小采集距离 (15cm)
4. **去重机制**：记录历史足迹避免重复

### 实现对比

| 方式 | 采集点数 | 数据质量 | 复杂度 |
|------|----------|----------|--------|
| 规划点触发 | 501 | 可能偏差 | 低 |
| **步态触发** | **90** | **准确** | **中** |

### 效果

- 过滤率: 82%
- 零重复采集点
- 步长分布: 0.28m - 1.36m

## DDR-006: 分片上传

### 上下文

采集的数据文件可能很大 (数百 MB 到数 GB)。单次上传大文件存在可靠性问题，网络中断需要重新上传。

### 决策

采用 AWS S3 **Multipart Upload** 分片上传机制。

### 原因

1. **可靠性**：单个分片失败只需重传该分片
2. **并发性**：多个分片可并行上传
3. **断点续传**：支持暂停和恢复
4. **大文件支持**：最大支持 5TB 文件

### 分片策略

| 文件大小 | 分片大小 | 并发数 |
|----------|----------|--------|
| < 100MB | 5MB | 1 |
| 100MB - 1GB | 10MB | 3 |
| 1GB - 5GB | 20MB | 5 |
| > 5GB | 50MB | 5 |

### 上传流程

```
1. CreateMultipartUpload → upload_id
2. UploadPart (part_number, data) → etag (循环)
3. CompleteMultipartUpload (all etags) → 完成
```

## DDR-007: 双模式支持

### 上下文

系统需要支持自动驾驶车辆和双足机器人两种不同的平台。

### 决策

设计统一的规划器接口，支持两种模式：

| 模式 | 状态维度 | 动作空间 |
|------|----------|----------|
| AUTO | 25维 | 4 离散动作 |
| HUMANOID | 43维 | 3 维连续速度命令 |

### 架构

```cpp
// 统一接口
class PlannerBase {
public:
    virtual Trajectory plan(const PlannerInput& input) = 0;
};

// 自动驾驶实现
class AutoPlanner : public PlannerBase { ... };

// 人形机器人实现（默认）
class HumanoidPlanner : public PlannerBase { ... };

// 工厂模式创建
class PlannerFactory {
    static std::unique_ptr<PlannerBase> create(const std::string& mode);
};
```

### 原因

1. **代码复用**：核心组件可共享
2. **独立优化**：各自模式可针对性优化
3. **易于扩展**：未来可添加新模式

## DDR-008: 配置热加载

### 上下文

生产环境中，修改参数需要重启服务会影响数据采集连续性。

### 决策

实现**配置文件监控**，支持热加载无需重启。

### 实现

```cpp
class ConfigWatcher {
public:
    void watch(const std::string& file_path,
               std::function<void()> callback);

private:
    void monitorThread();
    std::map<std::string, time_t> file_timestamps_;
};
```

### 监控配置

| 配置文件 | 监控内容 |
|----------|----------|
| `planner_weights.yaml` | RL 奖励权重 |
| `robot_data_collection.json` | 采集策略 |

## DDR-009: ROS2 通信

### 上下文

需要选择机器人软件通信框架。候选包括 ROS1、ROS2、自定义通信。

### 决策

采用 **ROS2 Humble** 作为通信框架。

### 原因

1. **实时性**：支持实时调度
2. **DDS**：底层使用 DDS，性能更好
3. **安全性**：内置安全机制
4. **生态**：丰富的机器人包

### 关键 Topic

| Topic | 类型 | 频率 |
|-------|------|------|
| `/robot/odom` | nav_msgs/Odometry | 50Hz |
| `/robot/joint_states` | sensor_msgs/JointState | 100Hz |
| `/plan/path` | nav_msgs/Path | 按需 |
| `/collection/trigger` | std_msgs/Bool | 按需 |

## DDR-010: 轻量级反馈机制

### 上下文

端云协同需要将采集结果反馈给云端用于重训练。完整数据量太大，需要设计轻量级反馈格式。

### 决策

设计 **DataPointMetadata** 轻量级元数据 (~20B)。

### 结构

```cpp
struct DataPointMetadata {
    Point position;           // 16B
    double timestamp;         // 8B
    SceneType scene_type;     // 1B
    double value_score;       // 8B
    // 总计 ~33B，压缩后 ~20B
};
```

### 对比

| 数据类型 | 大小 | 用途 |
|----------|------|------|
| 完整数据 | ~250B | 云端训练 |
| **元数据** | **~20B** | **边缘反馈** |
